## Сравнение с 3.4 (Eager Rebalancing)

| Характеристика | 3.4 Eager (`RangeAssignor`) | 3.5 Cooperative (`CooperativeStickyAssignor`) |
|----------------|---------------------------|----------------------------------------------|
| revokedPartitions у survivors | **Все** партиции revoked | **Ни одна** (`revokedPartitions: []`) |
| Rebalance durationMs | 158 мс / 22 мс | **0 мс / 0 мс** |
| Партиции survivors | Отозваны и переназначены заново | Остались на месте без прерывания |
| Партиция убитого | Переназначена после revoke всех | Инкрементально добавлена к survivor |
| Пауза обработки (session.timeout) | ~43 сек | ~43 сек (определяется session.timeout, не стратегией) |
| Lag после ребалансировки | 0 | 0 |
| Ошибки | 0 | 0 |

## Выводы

- **Cooperative rebalancing не отзывает партиции у survivors** — `revokedPartitions = []` у обоих. В отличие от eager (3.4), где все партиции были revoked у всех consumers.
- **Инкрементальное назначение:** consumer-1 получил partition 2 от убитого consumer-2 через `assignedPartitions: ["group-demo-topic-2"]` — только новая партиция, без потери существующей.
- **Пауза обработки** одинаковая (~43 сек) — она определяется `session.timeout.ms`, а не стратегией назначения. Cooperative ускоряет сам процесс ребалансировки, но не обнаружение мёртвого consumer.
- **В продакшене** cooperative выигрывает при частых ребалансировках (rolling deploy, autoscaling) — survivors не теряют свои партиции и продолжают обработку без пауз.
